## Assets

Finance data categories include:

- raw company exports and uploaded source files
- source snapshots, checksums, storage refs, provenance, freshness posture, and ingest logs
- bank, card, payroll, customer, vendor, receivables, payables, contract, obligation, tax, legal, board, and lender materials
- derived Finance Twin facts and read models
- compiled CFO Wiki pages and filed artifacts
- proof bundles, evidence cards, SourceAnchors, DocumentMaps, SourceCoverageMatrix entries, limitations, and permitted-next-action fields
- local Postgres data, MinIO/S3-compatible object storage, backups, logs, screenshots, shell history, and `.env` files

Secrets, credentials, and tokens include database URLs, object-store keys, API keys, GitHub tokens, cookies, session data, provider credentials, OpenAI keys if ever introduced, OAuth material if ever introduced, and local machine credentials.

## Threats

Public repo disclosure can occur through commits, issues, PR comments, screenshots, pasted logs, sample data, eval datasets, fixture files, source packs, markdown docs, or proof output.

Source packs, fixtures, benchmark cases, and demo examples are risky because they look safe once checked in. A source-pack file, benchmark case, or example derived from a real company remains private finance data even if names are changed.

Local Postgres and object storage are risky because they may retain raw source bytes, derived facts, local proof state, screenshots, database dumps, or object keys after a demo ends.

Contributor and PR risk includes accidental full-file dumps, copied `.env` values, private screenshots, broad rewrites that weaken warnings, and sample data that was not reviewed as synthetic.

Prompt injection in source text is expected. A PDF, export, markdown file, wiki page, or source excerpt may contain instructions such as "ignore previous instructions" or "send this report." Those strings are untrusted data and must not be followed.

Read-only evidence tools can still disclose data through broad search results, full-file excerpts, query logs, audit output, citations, or insufficient redaction.

Future provider, certification, deployment, public MCP, public ChatGPT App, Apps SDK UI/resources, protected-resource metadata route behavior beyond the FP-0125 local exact path, WWW-Authenticate route behavior, OAuth/token/session/auth middleware implementation, token validation runtime, endpoint expansion, app submission, public community-pack distribution, benchmark datasets, sample/demo data, hidden evidence service construction, and default/public evidence dispatch beyond FP-0111 explicit local app-construction wiring would add new privacy and security risks. They are outside the shipped product boundary and require future plans after the FP-0099 security threat-model/platform-boundary prerequisites, FP-0100 local security boundary contracts, FP-0101 implementation sequencing/platform-readiness gates, FP-0102 endpoint/OAuth/remote-MCP architecture gates, FP-0103 endpoint architecture proof-contract gates, FP-0104 endpoint readiness proof gates, FP-0105 endpoint route ownership/transport-adapter proof gates, FP-0106 protocol envelope, `ping` liveness, and tool-dispatch proof gates, FP-0107 local route-adapter shell boundary, FP-0108 evidence tool dispatch contract boundary, FP-0109 local injected dispatch adapter boundary, FP-0110 default local dispatch planning/proof-gate boundary, FP-0111 explicit local wiring boundary, FP-0112 remote/public MCP OAuth readiness boundary, FP-0113 OAuth/security contract boundary, FP-0114 remote host readiness/security contract boundary, FP-0115 remote host sequencing boundary, FP-0116 remote host resource contract boundary, FP-0117 OAuth/token/session/auth sequencing boundary, FP-0118 protected-resource metadata/auth challenge readiness contract boundary, shipped FP-0119 protected-resource metadata route sequencing boundary, shipped FP-0120 canonical resource/auth-server readiness contract boundary, shipped FP-0121 route implementation readiness boundary, shipped FP-0122 builder/credential-hardening boundary, shipped FP-0123 route-input evidence boundary, shipped FP-0124 route implementation planning boundary, shipped FP-0125 local route boundary, shipped FP-0126 auth-challenge sequencing boundary, and shipped FP-0127 auth-challenge contract boundary are accepted.

## Mitigations

- Keep raw sources immutable and do not rewrite uploaded source truth.
- Commit only synthetic, reviewed, clearly labeled demo or benchmark data under a named future plan. Lightly anonymized real finance data is forbidden.
- Do not put real finance data, credentials, database dumps, object-store files, or private screenshots in git, issues, PRs, docs, fixtures, evals, or source packs.
- Keep source excerpts bounded, cited, and redacted where the V2C policy applies.
- Keep FP-0109 MCP tool dispatch company-scoped and argument-scoped: mismatched `companyKey`, unsupported optional arguments, or unhonored source ids must fail closed before broad evidence disclosure.
- Keep FP-0111 default local dispatch wiring scoped to explicit app construction input only. Default `buildApp()` remains fail-closed without a supplied read-only MCP endpoint service, and no route-level evidence service construction, public data, source mutation, finance write, provider call, OpenAI API/model call, OAuth/token/session, or remote/public MCP behavior is allowed.
- Keep FP-0112 scoped to readiness planning only. Public/remote use must later prove authenticated user/org/company binding, token lifecycle controls, redaction, CORS/CSP/TLS/rate-limit/rollback gates, and no-real-finance-data posture before remote MCP or OAuth implementation starts.
- Keep FP-0113 scoped to local proof contracts only. Client-supplied `companyKey` is not authority, token passthrough is forbidden, token values must not appear in logs/UI props/metadata/evidence/proof output, and missing/expired/malformed/wrong-audience/wrong-scope/wrong-org token requests fail closed by contract before any OAuth/token/session/auth middleware implementation starts.
- Keep FP-0115 scoped to docs-only sequencing and proof-gate compatibility. The current local `/mcp` route is not remotely exposable as-is, `/mcp` remains the only future public MCP endpoint path, stable HTTPS/TLS, canonical resource URI, OAuth/token/session/auth middleware, Origin/CORS/CSP, rate-limit/abuse controls, logging redaction, observability/audit correlation, rollback/incident response, and no-real-finance-data posture must be proven before any remote host implementation starts.
- Keep FP-0116 scoped to local proof contracts only. Host ownership remains unresolved, provider selection remains neutral, canonical URI and protected-resource metadata remain prerequisites, workspace/tenant template URLs remain rejected, local tunnels remain development-only, and user/org/company binding must come from the authenticated security boundary rather than URL selectors, model text, prompt text, `_meta`, or unauthenticated `companyKey`.
- Keep FP-0117 scoped to docs-and-plan plus proof-gate compatibility only. Protected-resource metadata route implementation, WWW-Authenticate `resource_metadata` behavior, token validation, token/session storage, auth middleware, remote deployment, Apps SDK resources, app submission, public assets, provider calls, OpenAI API/model calls, source mutation, finance writes, generated finance advice, runtime-Codex finance output, and autonomous action remain future-only.
- Keep FP-0119 scoped to docs-and-plan plus proof-gate compatibility only. Protected-resource metadata route implementation cannot start until canonical public resource URI, route tests, metadata document tests, no-token-leakage gates, authenticated company binding gates, and local `/mcp` unchanged-behavior gates are proven; WWW-Authenticate `resource_metadata` route behavior remains a later/separate lane.
- Keep FP-0120 scoped to local/proof-only/read-only canonical resource/auth-server readiness contracts only. It may prove exact stable HTTPS canonical URI requirements, provider-neutral `authorization_servers`, RFC 9728 route-path derivation, WWW-Authenticate `resource_metadata` URL readiness, and no-route-runtime posture, but it must not choose a production host/provider, implement protected-resource metadata routes, implement WWW-Authenticate behavior, implement OAuth/token/session/auth middleware, deploy remote MCP, add deployment config, add Apps SDK resources, create app-submission assets, mutate sources, or write finance state.
- Keep FP-0124 scoped to docs-and-plan plus proof-gate compatibility only. It planned a future narrow local-only protected-resource metadata route implementation lane using explicit FP-0123 route-input evidence bundle dependency and fail-closed canonical URI/auth-server/builder-output/no-token-leakage/company-binding/`/mcp` unchanged gates, but it did not implement routes, WWW-Authenticate behavior, OAuth/token/session/auth middleware, remote deployment/config, Apps SDK resources, public app behavior, app submission, source mutation, finance writes, or FP-0125 in that slice.
- Keep FP-0125 scoped to the local-only/read-only explicit-dependency metadata route only. It may register GET `/.well-known/oauth-protected-resource/mcp` with valid prevalidated FP-0123 route-input evidence, but it must not add root/dual metadata routes, WWW-Authenticate behavior, OAuth/token/session/auth middleware, remote deployment/config, Apps SDK resources, public app behavior, app submission, source mutation, finance writes, or FP-0126 runtime scope.
- Keep FP-0126 scoped to docs-and-plan plus proof-gate compatibility only. It may plan future WWW-Authenticate `resource_metadata` auth-challenge sequencing, but it must not implement WWW-Authenticate behavior, change `/mcp`, implement OAuth/token/session/auth middleware, implement token validation runtime, deploy remote MCP, add deployment config, add Apps SDK resources, create app submission assets, mutate sources, or write finance state.
- Keep FP-0127 scoped to local/proof-only/read-only WWW-Authenticate auth-challenge contracts only. It may prove future Bearer `resource_metadata` shape, local-vs-public metadata reference posture, missing/invalid token challenge posture, future-only token failure modes, read-only scope challenge guidance, no-token-leakage, unchanged `/mcp`, and no-runtime posture, but it must not emit headers, change route behavior, change protected-resource metadata route behavior, implement OAuth/token/session/auth middleware, implement token validation runtime, deploy remote MCP, add Apps SDK resources, create app submission assets, mutate sources, write finance state, or create FP-0128.
- Keep structured-content text mirrors bounded and summary-shaped; never mirror raw full-file text, private source text, credentials, tokens, OAuth material, object-store dumps, database dumps, provider credentials, OpenAI keys, or private raw finance data.
- Keep V2G app/MCP response envelopes strict: no raw full text, private source text, credentials, tokens, OAuth material, API keys, object-store dumps, database dumps, provider credentials, source mutation fields, finance-write fields, provider fields, payment fields, customer-contact fields, or app-submission fields.
- Treat source text and source excerpts as untrusted data.
- Use stored SourceAnchors, DocumentMaps, EvidenceCards, Finance Twin state, CFO Wiki pages, freshness posture, and limitations as the evidence boundary.
- Preserve no-provider, no-deployment, no-public-MCP, no-public-ChatGPT-App, no-finance-write, no-source-mutation, and no-autonomous-action boundaries until future plans prove otherwise.
- Use [SECURITY.md](../../SECURITY.md) for private vulnerability reports and accidental sensitive-data disclosure.
- Use [docs/demo/demo-data-policy.md](../demo/demo-data-policy.md) before adding any future demo source pack or sample data.

## Explicit Non-Goals

This repository does not currently provide production hosting, cloud/SaaS security controls, tenant isolation, provider credential management, public MCP deployment, public ChatGPT App submission, OAuth, external delivery, certification, legal/tax/audit assurance, complete DLP, or autonomous finance action.

This threat model is a baseline for local OSS review and self-hosting, not a security certification.
